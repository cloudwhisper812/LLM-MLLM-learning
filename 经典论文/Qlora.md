在实际业务落地上，大家对 QLoRA 有一个普遍的认知：它是用来换空间的，不是用来提速的。在计算 $Y = X(W_0 + BA)$ 时，QLoRA 实际上需要先将 4-bit 的 $W_0$ 反量化（Dequantize） 回 BF16/FP16，然后再与输入 $X$ 进行矩阵乘法。这个反量化的过程带来了额外的计算开销（Compute Overhead）。因此，QLoRA 的训练速度往往比纯 BF16 的 LoRA 还要稍微慢一点，但它让你能在一张卡上跑起以前需要八张卡才能跑的模型。

 显存开销真相 (Memory Footprint)省下的部分：基础权重：通过 NF4 从 16-bit 压到 4-bit（7B 模型从 14GB $\rightarrow$ ~4GB）。优化器状态：由于冻结了 $W_0$，优化器只服务于极少量的 LoRA 参数，显存几乎忽略不计。管理开销：通过 Double Quantization (DQ) 将 Scaling Factors 从 0.5 bit/param 压到 0.127 bit/param。省不下的部分：激活值梯度 (Activation Gradients)：取决于 Batch Size / Seq Length / 模型维度。计算在 BF16 下进行，因此激活值依然是 BF16。计算 Buffer：反量化过程产生的临时显存占用。2. NF4 (NormalFloat 4) 的数学美学核心逻辑：权重服从 $N(0, \sigma^2)$，利用分位数（Quantile）让 16 个量化桶（Bin）承载相等的概率密度，达到信息论最优。工程 Hack：离线打表：不动态算分位数，而是针对标准正态分布预计算 16 个点。非对称零点：左边切 8 份，右边切 9 份，强制凑出一个绝对的 0.0 用于处理 Padding 等特殊值。3. Paged Optimizer (分页优化器)本质：利用 NVIDIA Unified Memory。机制：当 GPU 显存因长序列导致 Activation 暴增而 OOM 时，自动将暂不使用的优化器状态（Optimizer States）移出（Evict）到 CPU 内存；更新时再换回（Page back）。4. 技术本质：QLoRA 真的量化了 LoRA 吗？真相：QLoRA 并没有量化 LoRA 权重。皮肉关系：Q (NF4) 是“骨”：负责极低比特的模型存储。LoRA 是“皮”：负责在 BF16 空间提供高精度的残差修正。计算逻辑：$Y = X \cdot \text{dequant}(W_{NF4}) + X \cdot (B \cdot A)$。计算全在 BF16 进行，权重在计算瞬间“充气”回高精度，用完即丢。
